# RFC 贡献指南

感谢你为 ARTI CLI 贡献 RFC！

## 快速开始

### 1. 何时需要 RFC？

必须写 RFC：
- ✅ 添加新的 CLI 命令（如 `arti portfolio`）
- ✅ 修改现有命令行为（破坏性变更）
- ✅ 架构层面的调整（如数据源切换）
- ✅ API 接口变更（命令行参数、输出格式）
- ✅ 计费模型调整
- ✅ 重要依赖升级（如 Node.js 版本要求）

不需要 RFC：
- ❌ Bug 修复（除非涉及架构调整）
- ❌ 文档更新
- ❌ 代码重构（不改变外部行为）
- ❌ 依赖版本升级（无破坏性变更）
- ❌ 性能优化（不改变接口）

### 2. 创建 RFC

**方式 1：使用脚本（推荐）**

```bash
./scripts/create-rfc.sh
```

**方式 2：手动创建**

```bash
# 复制模板
cp rfcs/template.md rfcs/2026/RFC-2026-NNNN-your-title.md

# 编辑内容
code rfcs/2026/RFC-2026-NNNN-your-title.md

# 更新索引
# 在 rfcs/INDEX.md 中添加条目
```

### 3. 填写内容

按照模板结构填写：

#### 必填部分

- **元数据**：RFC 编号、标题、作者、状态、日期
- **摘要**：2-3 句话说清楚要做什么
- **动机**：为什么需要这个变更
- **详细设计**：如何实现
- **权衡与替代方案**：为什么选这个方案

#### 可选部分

- 风险评估
- 依赖关系
- 安全性考虑
- 性能影响
- 开放问题

### 4. 提交 RFC

```bash
# 1. 检查 RFC 完整性
# 确保必填部分都已填写

# 2. 更新 INDEX.md
# 将 RFC 添加到对应状态部分

# 3. 提交
git add rfcs/
git commit -m "RFC-2026-NNNN: 简短描述"
git push origin main

# 4. 创建 PR（可选）
# 大型变更建议先 PR 讨论
```

## RFC 生命周期

```
Draft → Proposed → Accepted → Implementing → Implemented
          ↓
      Rejected
```

### Draft（草稿）

- 初稿阶段，内容还在完善
- 可以边写边讨论
- 状态：`Draft`

### Proposed（已提出）

- 内容已完整，提交评审
- 团队成员在 `## 讨论记录` 中添加意见
- 状态：`Proposed`

### Accepted（已接受）

- 评审通过，准备实施
- 记录最终决策
- 状态：`Accepted`

### Implementing（实施中）

- 开始写代码
- 记录实施开始日期、负责人、分支
- 状态：`Implementing`

### Implemented（已实施）

- 功能已上线
- 填写实施记录（完成日期、PR、版本号、实际偏差）
- 状态：`Implemented`

### Rejected（已拒绝）

- 评审未通过
- 记录拒绝原因
- 状态：`Rejected`

## 写作技巧

### 摘要写作

✅ **好的摘要**
> 添加 `arti portfolio` 命令，支持用户管理投资组合，计算总收益率、风险指标和资产分布。

❌ **差的摘要**
> 这个 RFC 是关于组合的。

### 动机写作

✅ **好的动机**
> **问题陈述**: 用户需要手动在 Excel 中记录持仓，无法快速查看实时收益。
> 
> **用户故事**: 作为投资者，我希望在 CLI 中管理投资组合，以便随时查看实时收益。

❌ **差的动机**
> 别的工具都有，我们也要有。

### 权衡分析

✅ **好的权衡分析**
> **方案 A (选中)**: 本地 JSON 存储
> - 优点: 无需数据库，离线可用
> - 缺点: 无法跨设备同步
> 
> **方案 B (未选中)**: Backend 存储
> - 优点: 跨设备同步
> - 缺点: 离线不可用
> 
> **为何不选**: 当前用户主要单设备使用，跨设备同步不是高优先级。

❌ **差的权衡分析**
> 就用方案 A，因为简单。

## 评审指南

作为评审者，重点关注：

### 1. 完整性

- [ ] 摘要清晰？
- [ ] 动机充分？
- [ ] 设计详细？
- [ ] 权衡分析合理？

### 2. 可行性

- [ ] 技术方案可行吗？
- [ ] 风险可控吗？
- [ ] 资源够吗？

### 3. 影响范围

- [ ] 是否有破坏性变更？
- [ ] 迁移策略合理吗？
- [ ] 文档需要更新吗？

### 4. 长期维护

- [ ] 增加复杂度吗？
- [ ] 技术债可接受吗？
- [ ] 未来扩展性如何？

## 讨论规范

在 RFC 中添加讨论记录：

```markdown
## 讨论记录

### 2026-05-19 - Alice

建议将 `portfolio` 拆分为独立的子命令：
- `arti portfolio list`
- `arti portfolio add`
- `arti portfolio analyze`

这样更符合 CLI 惯例。

**决策**: 采纳，更新设计方案。

---

### 2026-05-20 - Bob

关于数据存储，建议使用 SQLite 而不是 JSON：
- 更好的查询性能
- 支持事务
- 更容易扩展

**决策**: 暂不采纳。理由：
1. 当前数据量小（< 100 条持仓）
2. JSON 更轻量，无需额外依赖
3. 未来可以迁移到 SQLite（不破坏接口）
```

## 常见问题

### Q: RFC 太长怎么办？

A: 如果 RFC 超过 500 行，考虑拆分：
- 拆分为多个独立的 RFC
- 使用 `assets/` 目录存放详细的技术设计
- 主 RFC 保留核心设计，链接到外部文档

### Q: RFC 需要更新怎么办？

A: 直接编辑 RFC 文件：
- 更新 `最后更新` 日期
- 在 `## 变更历史` 记录修改
- 如果是重大变更，考虑新建 RFC 并标记 `取代` 关系

### Q: RFC 被取代了怎么办？

A: 
- 将旧 RFC 状态改为 `Superseded`
- 在元数据中标记 `被取代: RFC-YYYY-NNNN`
- 新 RFC 中标记 `取代: RFC-YYYY-NNNN`

### Q: 需要保密的 RFC 怎么办？

A: 
- 在 RFC 文件头部添加 `**保密**: 内部文档`
- 不要在 GitHub 公开仓库中提交
- 使用内部 Git 仓库或文档系统
