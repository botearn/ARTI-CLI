## 一、高危

### H1. billing.ts `.data` 访问错误 —— 余额显示与扣费预检静默失效（真实运行时 bug）

- 位置：`src/billing.ts:290-291, 318`
- 问题：`supabaseSelect<T>` 返回 `{ data: T }`，第 280 行已解构出 `pricing` / `tiers` 两个数组，后续又对数组访问 `.data`。数组无此属性，运行时不抛错，`?? []` 静默兜成空数组。dist 产物行为相同，即线上版本亦如此。
- 后果：
  - `tierRow` 恒 `undefined` → `weeklyQuota = 0` → 周包恒显示 `0/0`，balance 丢失周包部分；
  - `limit5h = 0` → 5h 限流预检被 `limit5h > 0` 守卫跳过，完全失效；
  - `pricing` 恒 `[]` → `getFeatureCost` 恒返回 0 → `assertSufficientCredits` 永不拦截；credits 速查表全显示 `0 Credits ≈ $0.00 ✓`；RPC 扣费失败时报错文案变成"需要 0 Credits"。
- 修复：`tiers.data` → `tiers`，`pricing.data` → `pricing`（共 3 处）。补 `getActiveBillingState` 成功路径测试。
- 状态：✅ 已修复（2026-07-19）

### H2. 疑似双重扣费：chat / quick-scan（需后端确认）

- 位置：`src/commands/chat.ts:33`、`src/commands/product.ts:28`（`withBilling` → RPC `consume_credits_atomic`，`src/billing.ts:386-401`）
- 证据：v1-chat SSE 协议含 `billing` 事件，测试 fixture 为 `{"charged":true,"cost":1}`（`tests/apiEdgeV1.test.ts:42`）；v1-scan-stock envelope 带 `meta.billing.charged`；`src/api.ts:663` 注释自认"消费完成即确认计费"；RFC-2026-0005 设计为"计费展示读服务端 billing 元数据，替代 CLI 侧价格判断"。
- 后果：若 Edge 函数已记账，每次 `arti chat` / `arti quick-scan` 扣两份 Credits。
- 修复：与后端确认 v1-chat / v1-scan-stock 是否服务端计费；若是，移除 CLI 侧 `applyDeduction`（保留预检），改为读 billing 元数据展示。research/orchestrator 路径（Railway 后端）未见服务端计费证据，CLI 扣费应是唯一记账点。

### H3. orchestrator fallback 完全不扣费

- 位置：`src/commands/research.ts:1217, 1224-1280`
- 问题：`runOrchestrator` 的 catch 调 `runFallback`，并行跑 7 个分析师（7 次 `stock-research` 调用，服务器成本真实发生）并输出完整研报，全程未调 `applyDeduction`。
- 后果：orchestrator 越不稳定，用户越免费用。
- 修复：fallback 产出非空报告后按 `panorama`/`deepReport` 对应 featureKey 补扣，或明示免费策略。

### H4. 凭据文件 0644 落盘，本机任意用户可读（安全）

- 位置：`src/config.ts:209-212`（`saveConfig`）、`src/browser-login.ts:99-102`（`savePendingLogin`）
- 问题：`writeFileSync` 未指定 `mode`，`~/.config/arti/config.json`（含 access/refresh token）与 `pending-login.json`（含 `session_id` + `poll_token`）按默认 umask 落盘，实测 `-rw-r--r--`、目录 `0755`。
- 后果：refresh token 等同长期凭据，多用户机器/共享 agent 上可被直接读走；`pending-login.json` 泄露可被抢先轮询领走完整 token 对（`browser-login.ts:59-61`）。
- 修复：写文件 `{ mode: 0o600 }`、建目录 `{ mode: 0o700 }`；`loadConfig` 检测既有文件权限并收紧。

### H5. 命令失败退出码恒为 0，违反文档契约

- 位置：`src/commands/chat.ts:48-54`、`src/commands/product.ts:33-40`、`src/commands/research.ts:806-814`、`src/core/handler.ts:29-33`
- 问题：所有错误被 catch 后仅 `printError`，从不设 `process.exitCode`。实测未登录执行 `arti quick-scan AAPL` 退出码为 0。AGENTS.md 明文承诺"退出码非 0 表示失败"。
- 后果：agent 无法区分成败，可能解析空输出当成功，或盲目重试造成重复扣费。
- 修复：各命令 catch 分支设 `process.exitCode = 1`（积分不足可用专用码）。

### H6. `full`/`deep --json` stdout 被人类日志污染

- 位置：`src/commands/research.ts:942-1162`（125 处 `console.log`，无 `isJsonMode()` 守卫）
- 问题：`--json` 模式下 SSE 事件处理中的进度/结论全部直写 stdout，最后才打印 JSON，`arti deep AAPL --json | jq` 必失败。对比 `quick-scan` 输出是干净 JSON，三个命令行为不一致。
- 修复：JSON 模式下人类可读渲染全部跳过（或转 stderr）。

### H7. `checkForUpdate` 污染 --json stdout 且延迟进程退出

- 位置：`src/index.ts:287`（未 await 的浮动 promise）+ `src/update-check.ts:93-97`
- 实测：a) 缓存有新版本时 `arti whoami --json` stdout 在 JSON 前多一行"新版本可用"（已复现）；b) 浮动 promise 持有 fetch + 5s `setTimeout` 活跃句柄，每 24h 首次调用时进程要等 npm registry 请求 settle 才能退出（慢网/离线 >5s）。
- 修复：`--json` 模式不打印（或打 stderr）；定时器 `unref()`。

## 四、审计过、确认无问题的方面

- **Shell 注入**：全项目仅两处子进程调用，均无注入面（spawn 参数数组 / 固定字符串 execSync）。
- **TLS**：无禁用证书校验；默认 URL 全部 HTTPS（唯一 http 是本机 localhost）。
- **输入注入**：symbol 等只进 JSON body；poly slug 有 `encodeURIComponent`；`setConfigValue` 有 key 白名单 + `__proto__`/`constructor` 防护。
- **`--json` 敏感字段**：whoami token 已 mask；config list/get 已 mask；`arti token` 打印全量是文档化设计。
- **SSE 流处理**：buffer 切分、`[DONE]`、error 事件、finally 中 reader.cancel + abort + clearTimeout 均正确；提前退出会取消底层连接（有测试覆盖）。
- **callEdge/callBackend 重试**：仅网络错误/5xx、次数有上限、401 刷新不重入。
- **金额精度**：credits 全链路整数（Math.ceil），美元换算仅展示用，无浮点问题。
- **空结果不扣费**：chat 空回复、orchestrator 无产出均不扣，逻辑正确。
- **update-check 本身**：5s 超时、24h 缓存、静默失败、不发用户数据。
- **tracker**：只记录命令名 + symbol，无 PII。
- **Supabase anon key 硬编码**：设计公开的 publishable key，安全依赖服务端 RLS。
- **错误分类（网络层）**：DNS/ECONNREFUSED/timeout/401/429/5xx 映射准确。
- 无 `eval`/`new Function`、无 SQL 拼接。
